Refactoring Java, not again! Jun, 2011

Paul Graham very rightly says, a programming language should be malleable. It should allow you to quickly prototype a project and then beat the code into shape.

Static typing, like in java, is prohibitive of this approach. Another feature I feel is unnecessarily bureaucratic : interfaces. Interfaces sound too good to be true. They smell a lot like modularity and it is remarkable how if used correctly, they can lead to highly decoupled code. But interfaces also hide a lot of data. The actual implementation may contain substantial information that is inaccessible to outside code.

The more I program in Java, the more I feel it spends more time preventing me from doing things, than helping me out. In a way, you turn out to be your own "pointy-haired-boss". First, the PHB in you writes the interface. Then the employee in you starts writing an implementation based on the guidelines laid down by the PHB. Once this is done, you both are happy until there arises a requirement for some more features that this interface doesn't support. You send a request to PHB, he writes more of the interface and the employee in you writes more code to implement the interface and this cycle continues.

I am writing a RIA in GWT. I am trying to adhere to the design patterns laid out in the documentation. Sometimes, in the spirit of the code, just to make something work, I tend to step out of the design pattern and code arbitrarily. Now, when I have to refactor the code to align it to the pattern, I find myself terrified of the effort it would take to acheive that.

Having been on Java for about 3 months, when I try to write python code, I find myself thinking in Java. I find the lack of types and interfaces difficult to manage. I am writing up un-pythonic reflection based systems to implement features similar to interfaces and type-checking in java.

Python, I shall see you soon, and when I do...

A simple proxy servlet for GWT devmode May, 2011

Disclosure: The source code and some other ideas are from one of Kenneth Jorgensen's blog post

As you start developing with the Google Web Toolkit, you may find yourself stuck because of the "Same Origin Policy" restrictions that browser developers have implemented.

The SOP restricts scripts running on the browser from placing XmlHTTPRequests to hosts other than the host from which the current page was requested. This is a short, possibly imprecise definition. What really matters is that if you develop your server-side API in a language other than Java, you will have trouble accessing that API while in devmode.

What you can do is, use a language-agnostic servlet that runs on the GWT development mode server and proxies your requests to another server.

I won't dump the entire java servlet file here. You can find a link here. There is a dirty hack in the servlet; I describe it later.

To make this work, you have to edit the web.xml file within the WEB-INF directory within the war directory not the *.gwt.xml file. As far as I understand, (I am a noob to GWT) the *.gwt.xml file is to be updated with the details of a servlet when you use GWT-RPC.

Here is a snapshot of my web.xml.

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE web-app
    PUBLIC "-//Sun Microsystems, Inc.//DTD Web Application 2.3//EN"
    "http://java.sun.com/dtd/web-app_2_3.dtd">

<web-app>
    <servlet>
        <servlet-name>proxy</servlet-name>
        <servlet-class>in.dmulog.gwt.ProxyServlet</servlet-class>
    </servlet>
    <servlet-mapping>
        <servlet-name>proxy</servlet-name>
        <url-pattern>/proxy/*</url-pattern>
    </servlet-mapping>
    <!-- Default page to serve -->
    <welcome-file-list>
        <welcome-file>Client.html</welcome-file>
    </welcome-file-list>
</web-app>

Now, if you update the code according to your own setup, any requests at localhost:8888/proxy/this/uri will be forwarded to localhost:8000/this/uri. Or as the case may be. You can define all this in the java servlet file.

Now about the dirty hack I talked about. Within the java file, line number 68 reads:

uri.append(req.getRequestURI().substring(6));

What the substring part does is converts localhost:8888/proxy/foo/bar/baz to a request to localhost:8000/foo/bar/baz.

If we don't use the substring hack, it remains as a request to localhost:8000/proxy/foo/bar/baz which might not be useful in your case. You can possibly do this better with splitting the string and joining. But hey, I said it was dirty.

Happy coding.
